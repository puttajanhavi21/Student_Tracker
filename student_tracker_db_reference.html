Core design principles

🔑 Identity strategy
All primary keys use SERIAL (auto-increment integer). Surrogate PKs are used throughout — never natural keys like mahe_id or course_code as foreign keys. Natural keys appear as UNIQUE columns for lookups but not as FK targets.
SERIAL PKs Surrogate keys UNIQUE naturals
📐 Normalisation
The schema is in 3NF. Key decisions: (1) Course catalogue is separate from enrolments — course data doesn't repeat per student. (2) Grade components and assignments are separate tables — not columns on enrolment. (3) Labels and tags are separate entities with junction tables. (4) User preferences are separated from users to keep the core table lean. (5) Fitness goals are versioned with date ranges rather than overwritten.
🔗 Foreign key behaviour
ON DELETE CASCADE — used when child rows have no meaning without the parent (e.g., tasks, notes, mood logs belong to a user).
ON DELETE SET NULL — used when the child row can exist independently but loses its reference (e.g., a task's link to an assignment can be nulled without deleting the task).
ON DELETE RESTRICT — used to prevent accidental deletion of shared catalogue data (e.g., courses referenced by enrolments).
🏷️ ENUM vs CHECK
PostgreSQL CREATE TYPE ... AS ENUM is used for multi-table shared types (task_status, mood_type, grade_type). Inline CHECK constraints are used for single-table categorical fields (project status, theme preference) to avoid over-engineering the type system.
🔏 Polymorphic references
The reminders table uses a controlled polymorphic pattern — three nullable FK columns with a CHECK constraint enforcing exactly one is non-null. This avoids EAV anti-patterns while maintaining referential integrity per target type.
Pattern used: CHECK ( (event_id IS NOT NULL)::INT + (task_id IS NOT NULL)::INT + (assignment_id IS NOT NULL)::INT = 1 )
🕒 Timestamp convention
All timestamp columns use TIMESTAMP WITH TIME ZONE (timestamptz). PostgreSQL stores these as UTC internally and converts on retrieval. The application layer specifies user timezone (stored in user_preferences.timezone). created_at defaults to NOW(); updated_at should be maintained by application-level trigger or ORM hook.

Cross-domain optional links

Academic linkage pattern: tasks, events, reminders, and notes all carry an optional enrolment_id FK (SET NULL on delete). This enables context-aware features ("tasks related to DBMS") without requiring the link. It's optional coupling, not a hard dependency.
Task → assignment link: A task can optionally reference an assignment row. This allows the to-do system to surface academic deadlines automatically without a separate sync mechanism.

Academic tracking
Courses, enrolments, assignments, grade components, and attendance. The courses table is a shared catalogue — never duplicated per user.
Key decision: enrolments is the central join between users and courses. All academic data (assignments, grades, attendance) hangs off enrolment_id, not user_id + course_id. This avoids compound FK complexity and correctly models course-instance data.

reminders (polymorphic)

A single reminder table covers all entity types. Exactly one of event_id, task_id, assignment_id must be non-null — enforced by CHECK constraint.
CHECK ( (event_id IS NOT NULL)::INT + (task_id IS NOT NULL)::INT + (assignment_id IS NOT NULL)::INT = 1 )
Status lifecycle: pending → sent (background job fires notification) → dismissed (user action). The scheduler queries WHERE status = 'pending' AND remind_at <= NOW() — covered by a partial index.
Notes & personal projects
Hierarchical note folders, full-text searchable notes with tagging, and project tracking with milestones.

note_folders

Self-referential tree structure via parent_folder_id. Supports arbitrary nesting depth. UNIQUE(user_id, folder_name, parent_folder_id) prevents duplicate folder names within the same parent.
⚠️ For deep tree traversal, use PostgreSQL's WITH RECURSIVE CTEs. Do not model this as an adjacency list without recursive queries in mind.

notes

Indexed for full-text search via pg_trgm GIN indexes on both title and content columns. Supports fuzzy matching (ILIKE queries on trigrams).
CREATE INDEX idx_notes_title_trgm ON notes USING GIN (title gin_trgm_ops); CREATE INDEX idx_notes_content_trgm ON notes USING GIN (content gin_trgm_ops); -- Query pattern: SELECT * FROM notes WHERE user_id = 1 AND (title ILIKE '%normalization%' OR content ILIKE '%normalization%');

Fitness tracking
Daily logs, versioned goals, and individual workout sessions. The three-table design prevents the daily summary table from becoming overloaded.

fitness_daily_logs

One row per user per day. UNIQUE(user_id, log_date) enforced. All metrics are nullable — a user might only log steps without food.
Design choice: All daily aggregates live here (steps, calories, water, sleep, weight). Granular workout data goes in workout_sessions. This avoids recalculating daily totals on every query.

fitness_goals (versioned)

Goals are time-ranged: effective_from and nullable effective_to. This preserves history and allows future goal scheduling. A LATERAL subquery retrieves the currently active goal for any given date.
-- Get active step goal for a given date: SELECT target_value FROM fitness_goals WHERE user_id = 1 AND goal_type = 'steps' AND effective_from <= CURRENT_DATE AND (effective_to IS NULL OR effective_to >= CURRENT_DATE) ORDER BY effective_from DESC LIMIT 1;

Notifications & data export

notifications

Soft-polymorphic: source_table (VARCHAR) + source_id (INT) identify the originating entity without FK constraints. This is intentional — notifications are an audit/inbox layer that should not cascade-delete when source rows are removed.
Query pattern: The dashboard fetches unread count via WHERE user_id = ? AND is_read = FALSE, covered by a partial index. Mark-all-read uses a single UPDATE with timestamp.

export_requests

Schema support for PDF/Excel exports. Export jobs are queued asynchronously — this table tracks state (queued → processing → completed/failed) and the generated file URL (S3 or local storage). The application reads scope to determine what data to include.

Important queries
Production-ready SQL for the four core query patterns, plus bonus queries for fitness and mood trends.

Q1 — All tasks for a user (with labels)

SELECT t.task_id, t.title, t.status, t.priority, t.category, t.due_date, ARRAY_AGG(tl.label_name ORDER BY tl.label_name) FILTER (WHERE tl.label_name IS NOT NULL) AS labels FROM tasks t LEFT JOIN task_label_map tlm ON tlm.task_id = t.task_id LEFT JOIN task_labels tl ON tl.label_id = tlm.label_id WHERE t.user_id = 1 AND t.status != 'cancelled' GROUP BY t.task_id ORDER BY CASE t.priority WHEN 'critical' THEN 1 WHEN 'high' THEN 2 WHEN 'medium' THEN 3 ELSE 4 END, t.due_date ASC NULLS LAST;

Q2 — Dashboard summary (CTE pattern)

WITH task_stats AS ( SELECT COUNT(*) AS total_tasks, COUNT(*) FILTER (WHERE status = 'completed') AS completed_tasks, COUNT(*) FILTER (WHERE status = 'pending' AND due_date < NOW()) AS overdue_tasks FROM tasks WHERE user_id = 1 ), course_stats AS ( SELECT COUNT(*) AS total_courses, ROUND(AVG(grade_points) FILTER (WHERE grade_points IS NOT NULL), 2) AS gpa FROM enrolments e JOIN courses c ON c.course_id = e.course_id WHERE e.user_id = 1 AND e.academic_year = 2024 ), fitness_today AS ( SELECT steps, calories_burned, water_ml FROM fitness_daily_logs WHERE user_id = 1 AND log_date = CURRENT_DATE ) SELECT ts.*, cs.*, ft.* FROM task_stats ts CROSS JOIN course_stats cs LEFT JOIN fitness_today ft ON TRUE;

Q3 — Upcoming events + reminders (next 7 days)

SELECT 'event' AS item_type, e.title, e.starts_at AS due_at FROM events e WHERE e.user_id = 1 AND e.starts_at BETWEEN NOW() AND NOW() + INTERVAL '7 days' UNION ALL SELECT 'reminder', COALESCE(ev.title, t.title, a.title), r.remind_at FROM reminders r LEFT JOIN events ev ON ev.event_id = r.event_id LEFT JOIN tasks t ON t.task_id = r.task_id LEFT JOIN assignments a ON a.assignment_id = r.assignment_id WHERE r.user_id = 1 AND r.status = 'pending' AND r.remind_at BETWEEN NOW() AND NOW() + INTERVAL '7 days' ORDER BY due_at ASC;

Q4 — Course performance summary

SELECT c.course_code, c.course_name, c.credits, e.final_grade, e.grade_points, COUNT(DISTINCT a.assignment_id) AS total_assignments, COUNT(DISTINCT a.assignment_id) FILTER (WHERE a.is_submitted) AS submitted, ROUND(AVG(a.obtained_marks / NULLIF(a.max_marks,0) * 100), 1) AS avg_assignment_pct, ROUND( COUNT(ar.record_id) FILTER (WHERE ar.is_present)::NUMERIC / NULLIF(COUNT(ar.record_id), 0) * 100, 1 ) AS attendance_pct FROM enrolments e JOIN courses c ON c.course_id = e.course_id LEFT JOIN assignments a ON a.enrolment_id = e.enrolment_id LEFT JOIN attendance_records ar ON ar.enrolment_id = e.enrolment_id WHERE e.user_id = 1 AND e.academic_year = 2024 AND e.semester = 5 GROUP BY c.course_id, e.enrolment_id ORDER BY c.course_name;

Bottlenecks & scalability notes
Honest analysis of where this schema will strain under load, and what to do about it.

Identified bottlenecks

⚡ Dashboard query (Q2) — fan-out CTEs
The dashboard runs 4–5 subqueries on every page load. At scale (>100k users), this becomes expensive. Mitigation: materialise the dashboard row in Redis (TTL 60s). Invalidate on task/fitness/grade write. The schema supports this because all source tables have clear created_at/updated_at audit columns.
⚡ Reminder scheduler hot path
A background cron that polls reminders WHERE status='pending' AND remind_at <= NOW() will hammer this table at scale. Mitigation: The partial index on status='pending' helps. At high scale, migrate to a proper job queue (pg_cron, BullMQ, or Temporal) and only scan rows in the next N minutes, not all pending.
⚡ attendance_records growth
At 1,000 students × 5 courses × 100 classes = 500k rows/semester. Over 4 years: ~2M rows. Mitigation: Partition by academic_year using PostgreSQL range partitioning. Old semesters become read-only, hot semesters stay small.
CREATE TABLE attendance_records_2024 PARTITION OF attendance_records FOR VALUES FROM (2024) TO (2025); -- if partitioned by year
⚡ notifications table — unbounded growth
Notifications accumulate indefinitely. Mitigation: Add a scheduled job to archive/delete notifications older than 90 days where is_read=TRUE. Or use a expires_at column and filter on it.
⚡ updated_at maintenance
PostgreSQL does not auto-update updated_at columns. Mitigation: Create a reusable trigger function and attach it to every table that has updated_at.
CREATE OR REPLACE FUNCTION set_updated_at() RETURNS TRIGGER AS $$ BEGIN NEW.updated_at = NOW(); RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trg_tasks_updated_at BEFORE UPDATE ON tasks FOR EACH ROW EXECUTE FUNCTION set_updated_at(); -- Repeat for: notes, events, projects, fitness_daily_logs, users, etc.
✅ Scalability strengths
— All high-cardinality queries are indexed on (user_id + discriminator) composites, limiting scans to per-user data.
— No computed columns stored in the DB (attendance %, GPA, task completion %) — these are always calculated at query time or cached in application layer.
— ENUM types are schema-level, avoiding expensive string comparisons on status columns.
— The schema is connection-pool friendly — no session-local state, all queries are pure reads/writes.
